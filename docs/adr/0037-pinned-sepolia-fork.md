# ADR-0037: Pinned Sepolia fork for cross-environment determinism

**Status:** Accepted
**Date:** 2026-04-19
**Related:** ADR-0027, ADR-0030, ADR-0032

## Context

EFS runs in several environments that all need to agree on contract addresses and EAS schema UIDs:

- **Local hardhat** — forked Sepolia for development (`yarn fork` / `yarn preview`).
- **Devnet VPS** at `https://178.104.79.94.nip.io` — the same forked-Sepolia setup as local, reset every ~7 days from `main`.
- **CI** — the same fork, running tests.
- **Production Vite client** (`github.com/efs-project/client`, statically hosted) — bundles a compile-time manifest of contract addresses + schema UIDs; users point their browser at it and it talks to whichever EFS chain it can reach (local → devnet → eventually Sepolia / mainnet).

All of these derive from the same EFS commit. If any of them diverge on contract addresses or schema UIDs, the client fails silently at runtime: it sends a call to address X expecting EFSIndexer and gets nothing back, or submits an attestation under schema UID Y that no one else indexes.

Contract addresses are set by the `CREATE` opcode: `address = keccak256(rlp([deployer, deployer_nonce]))`. That's already deterministic given a fixed deployer key and a known deploy order — and both are fixed in this repo. Schema UIDs are `keccak256(schemaString ++ resolverAddress ++ revocable)`; resolver addresses are deterministic by the same CREATE rule, so schema UIDs are too.

What is **not** deterministic is the Sepolia state underneath the fork at the moment `yarn fork` runs. Two environments forking Sepolia on different days see:

- Different blocks of history, which doesn't usually matter.
- Different `SchemaRegistry` contents — someone else may have registered a schema between the two fork times, changing which of our deploy calls hit `AlreadyExists`.
- Different EAS attestation history, which a test or simulate script might inspect.
- Different baseFee / gas dynamics that can change deployer nonce if any preflight call fails and retries.

Any of those can — at the margin — perturb our deployer nonce by one, which then shifts every subsequent contract address, which then shifts every schema UID that hashes a resolver address. A one-step nonce divergence between local and devnet is enough to silently break the client.

Separately: RPC rate limits matter. `yarn fork` without a pin pulls whatever state the fork needs to answer each call, on demand, hitting Alchemy's Sepolia free tier. Multiple fresh forks a day, across parallel agents, pile up requests.

## Decision

Pin the Sepolia fork to a specific block. Default set in `packages/hardhat/hardhat.config.ts`:

```ts
forking: {
  url: `https://eth-sepolia.g.alchemy.com/v2/${providerApiKey}`,
  blockNumber: Number(process.env.FORK_BLOCK ?? 10_691_000),
  enabled: process.env.MAINNET_FORKING_ENABLED === "true",
},
```

`yarn preview` (the one-command dev-stack launcher) sets `MAINNET_FORKING_ENABLED=true` in the fork child process env so the pin applies automatically. `yarn fork` run by itself requires the variable set in `.env` or the shell — documented in `packages/hardhat/.env.example`.

Commit `packages/nextjs/contracts/deployedContracts.ts` (regenerated by `yarn deploy` against the pinned fork) alongside any contract change. **This file is the contract-address source of truth across every environment.**

The resulting coordination unit across local / devnet / client is **the EFS commit SHA**:

- Devnet agent checks out commit `abc123`, runs `yarn deploy` → gets the same addresses committed in `abc123`'s `deployedContracts.ts`.
- Vite client built from `abc123` bundles the same addresses → reaches local or devnet interchangeably.
- Bumping `FORK_BLOCK` is the same kind of change as any other contract change: it requires regenerating and committing `deployedContracts.ts` in the same commit.

Pin advancement cadence is ad hoc — bump when we need state the current pin doesn't have (a newly registered schema on Sepolia, a newly deployed reference contract we want forked alongside ours), or when the pin is old enough that we're worried about RPC providers pruning archive state for that height. In either case: one commit, regenerate deployedContracts.ts, push, and every downstream environment rebases on that commit.

## Consequences

- **Local / devnet / client stay byte-identical on addresses and schema UIDs**, as long as they share the commit. The silent-divergence failure mode is eliminated.
- **RPC use drops dramatically.** Hardhat caches pinned-fork state in `~/.cache/hardhat-nodejs/`; subsequent `yarn fork` runs at the same pin hit Alchemy only for state not already cached. Parallel agents / repeated CI runs reuse the same cache.
- **`FORK_BLOCK` is the one-line coordination knob.** Overridable per-environment via env var; default lives in `hardhat.config.ts` so no configuration is needed for the happy path.
- **RPC archive pruning is the only eventual failure mode.** Some RPC providers drop pre-finalized or very old state. Alchemy's free tier currently supports full archive access back to Sepolia genesis, but that's a vendor choice, not a guarantee. When we bump the pin we'd want to check that the new block is still retrievable from every RPC we care about. Tracked as operational concern, not an ADR-level risk — the pin is trivially bumpable.
- **Bumping the pin is a real change, not a config knob.** It shifts every `deployedContracts.ts` entry, which forces a client rebuild and devnet re-deploy. Treat pin bumps the same as contract changes: rare, deliberate, in a commit.
- **Devnet resets stay cheap.** The devnet's 7-day reset cadence just re-deploys from whatever commit `main` is pinned to. Users running a client built against that commit keep working.
- **No impact on mainnet (ADR-0030).** Mainnet contracts are deployed directly against live mainnet state, not against a fork. This decision governs only the dev/devnet path.
- **Testing reproducibility improves.** Any test that inspects Sepolia state (existing schemas, reference attestations) is now deterministic against the pin.

## Alternatives considered

- **No pin — fork at latest block each run.** Rejected. Contract addresses mostly come out the same (CREATE is deterministic), but even a single nonce-perturbing edge case silently breaks the client on one environment while the other still works. Hard to debug after the fact; impossible to prevent without a pin.
- **CREATE2 with a fixed salt for every deploy.** Would make addresses truly independent of nonce ordering, but at the cost of every deploy script change, every test helper change, and an awkward pattern against a code base that doesn't otherwise need CREATE2. The pin achieves the same determinism with one line of config.
- **Snapshot the fork state to a file and ship it in-repo.** Overkill. Hardhat already caches fork state in `~/.cache/hardhat-nodejs/`; the first `yarn fork` at a new pin fills the cache, and the cache is hundreds of MB — not something to check in. Pinning solves the coordination problem without needing a snapshot artifact.
- **Point everyone at the devnet VPS directly instead of each running a local fork.** Rejected for dev ergonomics — single point of failure, round-trip latency, and the local fork is the right environment for iterating on contract changes before they land on devnet.
- **Use a different L2 testnet with deterministic-by-default behavior (e.g. a chain that doesn't require forking to get EAS).** Would require deploying EAS ourselves there, which is a bigger coordination problem than pinning a fork. Also, ADR-0032 commits to EAS as the foundational dependency and keeps us close to the canonical EAS deployments.
