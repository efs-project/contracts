# ADR-0053: SystemAccount ŌĆö the neutral system write-identity

**Status:** Proposed
**Date:** 2026-06-11
**Permanence-tier:** Durable (records an Etched-adjacent decision ŌĆö the address baked into canonical writes is Etched-at-first-write)
**Related:** ADR-0016 (deployer fallback ŌĆö formalized/replaced here), ADR-0039 (default-lenses chain ŌĆö `system` becomes its tail), ADR-0033 (kernel auto-attests system TAGs from its own address), ADR-0030 + ADR-0048 (permanence + burn-to-immutable, same model), ADR-0011 (transport anchors it authors), ADR-0052 (ValueRegistry / intern registry ŌĆö a future module). **Freeze-adjacent ŌĆö no schema change, but a deploy-ordering decision that must be settled before the Sepolia deploy.**

## Context

EFS already has two implicit "system" identities, both accidental. The **deployer EOA** is the final-fallback lens (ADR-0016/0039) and the on-chain author of the root namespace + `/transports/*` scaffolding. The **kernel's own address** is an attester too ŌĆö EFSIndexer auto-attests system `/tags/schema` TAGs from itself (ADR-0033). Neither was *chosen* as the system author; each is an artifact of which contract happened to call `EAS.attest`, or of a throwaway key.

A permanent filesystem needs a deliberate, neutral, code-governed identity to author the technically-necessary structure ŌĆö not a throwaway key, not an accident of attestation. This ADR formalizes that identity. It changes no schema; it is purely about *who* authors the canonical bootstrap and *what address* sits at the tail of the default lens chain.

## Decision

Introduce **`SystemAccount`** ŌĆö a thin, CREATE3-deterministic, code-governed *attester* contract. The lens label users see is **`system`**.

- **Write-through model.** In EAS the attester is the caller of `EAS.attest`. Official EFS contracts (modules) call `SystemAccount.attest(...)`, so the attester of all official/canonical data is `SystemAccount`. The contract is a thin relay ŌĆö `attest` / `revoke` / `multiAttest`, plus minimal typed sugar like `registerAnchor` ŌĆö **not a data store.**
- **Two authorities, only one human-held.** *Membership authority* = which contracts may write through it (an `authorizedModule` mapping; `setModuleAuthorization` is owner-only ŌĆö the EFS.eth Safe ŌĆö and **pre-burn only**). *Content authority* = WHAT each authorized module writes, which lives entirely in that module's own code. **No human can make `SystemAccount` emit a specific attestation.** Humans choose programs, never payloads. This is the *same* trust already granted the Safe via proxy-upgrade authority (ADR-0030/0048) and is closed by the same burn ŌĆö it is **not a new trust hole.**
  - **Owner write authority is confined to the one-time `bootstrap` ceremony, sealed at deploy (PR #24 P1 fix).** The general relay (`attest` / `multiAttest` / `revoke` / `registerAnchor`) is `onlyAuthorizedModule` ŌĆö *not* owner. The owner's only write power is the one-time `bootstrap(...)` (gated `onlyOwner` + `whenNotSealed`), which the deploy permanently locks via `seal()` after authoring the scaffolding and before/at ownership transfer to the Safe. So in steady state the owner (the Safe, a human multisig) can **not** emit or revoke arbitrary payloads as the permanent `system` attester ŌĆö preserving "humans choose programs, never payloads." Ordering: deploy ŌåÆ wire ŌåÆ register ŌåÆ bootstrap ŌåÆ **seal** ŌåÆ transfer.
- **As-if-immutable / burn.** Designed as though its behavior is permanent; the proxy is a dev crutch. Upgradeable pre-burn (fix bugs, authorize newly-designed official modules); frozen at burn (`ProxyAdmin.renounceOwnership`), the same model as the resolvers (ADR-0030/0048).
- **Core + satellites, not a monolith.** The permanent thing is the *identity (address)*. New official behavior is added by authorizing a *new module*, never by mutating `SystemAccount`. The authorization mapping **is** the extensibility seam.
- **What it authors (technically-necessary structure):** the bootstrap namespace (root, `/transports/`, `/config/` skeleton, reserved-key anchors like `contentType` / `name`) and canonical transport definitions (ADR-0011). Today the deployer EOA authors root + `/transports/*` in the deploy; this moves authorship to `SystemAccount` so the permanent root structure has a neutral, stable, code-defined author. Future official registries (name / contract-address, schema aliases) are deferred modules.
- **`ValueRegistry` is a future module** (the ADR-0052 intern registry). Interned PROPERTY values minted *through* `SystemAccount` are attested by `system` ŌĆö the faithful on-chain encoding of "nobody owns the value" ŌĆö which lets `ValueRegistry` stay redeployable while the canonical value author stays stable. Deferred per ADR-0052; the seam is what's built now.
- **Lens placement.** `SystemAccount` is the **last entry in the default lens chain** (`you ŌåÆ viewed-address ŌåÆ your-web-of-trust ŌåÆ system ŌåÆ [future: public-shared]`), **default-on but user-removable**. Because it is strictly last, first-attester-wins (ADR-0031) makes it purely **additive** ŌĆö it can never override a user's content, only fill gaps. The additivity guarantee holds because `system` is **removable but not reorderable** ŌĆö a user removes it (and loses the fallback) but can never move it ahead of their own content, so it can only ever fill gaps, never shadow a user's own attestation. It **formalizes and replaces** the ADR-0016/0039 deployer fallback. EFS.eth is **just-another-user**: opinionated/curated data is a normal opt-in lens (web-of-trust tier) with no special system power.
- **Structural vs lens-scoped.** `SystemAccount` creates *structural anchors* ŌĆö the namespace skeleton ŌĆö which exist for everyone regardless of lens (ANCHOR is non-revocable; the path walk is attester-agnostic). These can't be "turned off." It also makes *lens-scoped claims* (fallback PROPERTY values, official mappings) filtered to its lens ŌĆö these are what "turning system off" removes. Turning it off never breaks a user's own resolution (they are first in the chain); worst case is an `application/octet-stream` MIME fallback.
- **Naming.** `SystemAccount` (lens label `system`). Avoid "EFS" in the permanent name ŌĆö the project may rename, and permanent on-chain identities shouldn't encode the mutable project name. `Root` / `Kernel` / bare `System` rejected: they collide with the root anchor, with EFSIndexer's informal nickname, and are too generic.

## Consequences

- **No schema change; not in any schema UID.** `SystemAccount` is an *attester*, not a resolver ŌĆö it is in NO schema's `keccak256(fieldString, resolver, revocable)` tuple. It does **not** touch the 9 frozen schema UIDs and does **not block the freeze** (`docs/SEPOLIA_FREEZE_TABLE.md`).
- **But its address is Etched at first canonical write.** That address is baked into every canonical attestation it authors and is the default-lens tail, so it must be deployed **deterministically** (committed CreateX salt) as part of the frozen deploy set and **never moved** ŌĆö a redeploy forks official-data authorship.
- **Deploy-ceremony change.** `SystemAccount` deploys with the resolvers, authors the bootstrap scaffolding (replacing the deployer EOA), and its ownership transfers to the Safe. The default-lens fallback (router + client) points at `SystemAccount` instead of the deployer EOA ŌĆö a redeployable-view / client-config change (ADR-0039), not a frozen-schema one. **This must be settled before the Sepolia deploy** ŌĆö it is the one deploy-ordering decision the freeze introduces.
- **Closes the ADR-0016 wart.** The "deployer becomes a quasi-special address" concern flagged in ADR-0016 is resolved: the permanent system author is now a code-governed identity, not a throwaway EOA.
- **The kernel's own self-attested auto-TAGs (ADR-0033) remain kernel-authored and are NOT migrated to `SystemAccount`.** The kernel attests those system `/tags/schema` TAGs *inside its own `onAttest`* and cannot route through `SystemAccount` without a re-architecture; `SystemAccount` formalizes the *deployer-fallback* role (root/`/transports/*` bootstrap authorship + the default-lens tail), not the kernel's auto-tag identity. The two system identities (kernel-as-attester, `SystemAccount`-as-attester) stay distinct.

## Alternatives considered

- **Monolithic system contract.** Rejected: fuses the permanent identity with a churning feature set, maximizing upgrade blast radius on the contract you least want to touch. Core + satellites keeps the permanent thing tiny.
- **Per-official-contract identities.** Rejected: the default lens chain would have to enumerate N official addresses and re-enumerate per new module. One address is one stable terminal lens and a one-line trust rule ŌĆö "official = attested by `system`."
- **Deployer-EOA-as-permanent-system (status quo).** Rejected: a throwaway key as the permanent author of the root namespace is fragile ŌĆö the ADR-0016 wart, now closed.
- **Build the name / schema / address registries + a module/permission framework now.** Deferred (YAGNI): the authorization mapping is the only seam needed. Each registry is its own later ADR; building a framework for undesigned modules is the speculative-abstraction trap (cf. the rejected ADR-0045). The minimal `SystemAccount` ŌĆö identity + bootstrap authorship + last-lens + auth seam ŌĆö forecloses nothing.
