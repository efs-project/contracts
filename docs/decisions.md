### 2026-06-15 — [claude-opus-4.8 · dev] Sanitize the MIRROR uri in the external-body header (extends ADR-0024)
Codex P2 (PR #24): `EFSRouter.request()` interpolated the MIRROR `uri` verbatim into the `message/external-body; … URL="…"` Content-Type, sanitizing only `contentType`. MirrorResolver validates a MIRROR uri's scheme + length (ADR-0023) but not its body, so an attester-controlled uri carrying `"`, control bytes, or `\` could break the URL quoted-string and inject header parameters into every client served through that lens (worsened by this PR widening the off-chain allowlist). Fix: run the uri through the existing `_sanitizeHeaderValue` (strips `"`/`\`/<0x20) before building the header — the same model as ADR-0024. A well-formed URI cannot contain those bytes raw (RFC 3986 percent-encoding), so this only ever degrades a hostile uri to a broken redirect, never an injection. Read-path fix (covers already-stored MIRRORs); no schema/ABI change. Treated as a bug fix within ADR-0024's existing invariant, not a new ADR. Regression test in `EFSRouter.test.ts`.

### 2026-06-10 — [claude-opus-4.8 · dev] Canonical anchor-name encoding = NFC + percent-encode (uppercase hex)
Defined the canonical on-chain encoding for anchor `name` (ADR-0048, supersedes ADR-0025's reject-only rule), landing pre-freeze so it fixes the Schelling-point form of every folder/file name. Client encodes `NFC(human name)` then percent-encodes the reserved byte set (`%XX`, UPPERCASE hex); reserved set = C0 controls + DEL + space + `%` + `" # & / : = ? @ [ \ ] ^ \` { | }`. `EFSIndexer._isValidAnchorName` (single byte-pass) now enforces the byte-level canonical form: accepts unreserved bytes + high-bit UTF-8 + canonical `%XX` escapes; rejects bare reserved bytes, malformed/truncated escapes (`%`, `%2`, `%ZZ`), and lowercase-hex escapes (`%2f`) so each byte has exactly one encoding. NFC is **client-enforced** — too large for Solidity, so the resolver validates only percent-encoding/hex-case, not normalization. The `""`/`.`/`..` rejections from ADR-0025 are preserved. `"Q&A: Episode 5"` → `Q%26A%3A%20Episode%205`. Logic-only change (ANCHOR field string `"string name, bytes32 schemaUID"` unchanged); safe behind the EFSIndexer proxy. No seed/test re-encoding was needed — all existing anchor names (`docs`, `readme.txt`, persona names are PROPERTY *values* not anchor names) are reserved-char-free.

### 2026-05-31 — [claude-opus-4.6 · dev] LIST_ENTRY → pure identity; order + label as PROPERTYs (ADR-0046)

Resolves the OPEN free-text question below. Rather than add a `string` label field to LIST_ENTRY, we **removed `int256 weight`** from the schema entirely (LIST_ENTRY is now `bytes32 listUID, bytes32 target` — pure membership). Order *and* free-text label became PIN-bound (cardinality-1) PROPERTYs on the **stable entry UID** (`"weight"` = decimal-string rank, `"name"` = arbitrary-length label). One move fixes both open issues: reorder/edit re-PIN a PROPERTY in O(1) without churning the entry UID (so labels survive — the §7 orphan footgun is gone), and the 31-byte free-text cap disappears (label is a `string` PROPERTY; ANY-mode `target` is now `keccak(text)`). Sorting is client-side over the order PROPERTY; the on-chain SortOverlay was rejected because it can't source list entries without coupling the kernel to list semantics (a hardened invariant). Adversarially reviewed (PROCEED-WITH-CHANGES; the cheaper per-list ordering-vector alternative is recorded + rejected in the ADR). Validated on the deployed stack: Lists 56/56, simulate 42/42, client typecheck + unit 23/23, property round-trip 10/10 (80-byte label, lens isolation, reorder/edit keep UID). Full ADR: **ADR-0046**.

### 2026-05-30 — [claude-opus-4.8 · dev] Lists corrected to ADR-0044 (placed like files) + on-chain lens index

The earlier "lists in folder" work diverged from ADR-0044 §1 (it placed a list via `LIST.refUID = folder` + a stateful `ListResolver` parent index, and added a non-ADR `string name` field to the LIST schema). Reverted to the ADR design and corrected in the client: **a list is placed exactly like a file** — `ANCHOR(anchorType=LIST_SCHEMA_UID) + LIST(free-floating config) + PIN(definition=anchor, refUID=LIST)`. It surfaces through the existing `getDirectoryPageBySchemaAndAddressList(parent, LIST_SCHEMA_UID, …)`, opens by resolving `EdgeResolver.getActivePinTarget(anchor, attester, LIST_SCHEMA_UID)`, and **deletes by revoking the placement PIN** (anchor + LIST stay, like a file's DATA). The list's name is the anchor name. No new contract code was needed — `EdgeResolver` already supports `PIN → LIST` (a code-review subagent confirmed the placement model is sound).

**Editions/lenses are on-chain, not logs.** Each contributor (entry-attester) is an "edition" of a list. Discovery was briefly done via `ListEntryAttested` getLogs, but logs aren't consensus state and smart contracts can't read them — so `ListEntryResolver` now keeps an append-only per-list attester index (`_listAttesters`, ADR-0009) exposed as **paginated** `getListAttesters(listUID, start, len)` + `getListAttesterCount`. This un-defers ADR-0044's "on-chain reverse-lookup index" (DEFERRED list). Pagination was locked in before the mainnet freeze (ADR-0044 §8) because an unbounded `address[]` return is O(N²) on memory and can OOG an `eth_call`; active lenses = those with `getLength > 0`. The UI's edition picker reads this view and defaults to the curator's edition; you edit only your own. Changing `ListEntryResolver` storage shifts the `LIST_ENTRY` schema UID (ADR-0044 §8) — free pre-launch.

**RESOLVED 2026-05-31 by ADR-0046** (see entry above): the limit was lifted not via a `string` field but by removing `weight` from the schema and moving order *and* label to PROPERTYs on the stable entry UID. Browser-verified end-to-end (pre-ADR-0046): create/open/add/reorder/**delete**/editions across two contributors. See `docs/superpowers/plans/2026-05-30-lists-adr0044-correction.md`.

### 2026-05-29 — [claude-opus-4.8 · dev] ANY-mode list items pack UTF-8 text directly into the entry `target` (not keccak)

**Superseded 2026-05-31 by ADR-0046:** the bytes32-packing (and its 31-byte cap) is gone. ANY-mode `target` is now `keccak(text)` and the human-readable text is an arbitrary-length `name` PROPERTY on the entry UID, so it is still fully on-chain and cross-device — without the length cap. The production client (`efs-project/client`) must read the label from the `name` PROPERTY, not by unpacking `target`. Original entry kept below for history.

ADR-0044 §2 suggested intrinsic/ANY-mode list items use a member key of `keccak256(abi.encode("efs-list-intrinsic", payload))`. That convention stores only a hash on-chain — the human-readable text ("Milk") lives nowhere on-chain, so a grocery/todo list rendered on another device or wallet shows opaque hashes. We instead encode ANY-mode items by **packing the UTF-8 bytes of the item text directly into the bytes32 `target`, right-padded (Solidity bytes32-string idiom), capped at 31 bytes.** The text is then fully on-chain, cross-device, and reversible for display; dedup still works (identity key = the text bytes, so duplicates collide exactly). This is a **client-convention choice only** — the `ListEntryResolver` enforces just `target != 0` for ANY mode, so no schema/resolver change was needed and ADR-0044 is not superseded (the keccak form was a suggestion, not an enforced invariant). Tradeoff: items > 31 bytes are rejected at the UI; longer labels would use the ADR-0044 §7 PROPERTY-on-entry pattern (deferred). The production client (`efs-project/client`) must use this same packing to read entries this repo writes. Implemented in `packages/nextjs/components/explorer/ListPreviewPane.tsx` (`packText`/`unpackText`). Ordering uses the entry `int256 weight` as an on-chain rank (spaced by 1e6; drag-reorder = revoke + re-attest the moved item with a midpoint weight — a real transaction, surfaced in the background-ops drawer). Architect + code-review subagents consulted; reorder neighbor-index off-by-one and two missing re-entrancy (`busy`) guards were caught and fixed pre-commit.

### 2026-04-20 — [claude] `yarn preview` now group-kills its children on shutdown

Problem: every `yarn preview` cycle was leaking a `hardhat node` process. User accumulated 7 stale forks over a day, enough to spin the fans visibly. Root cause: `scripts/claude-preview-launch.mjs` spawned `yarn hardhat:fork` and `yarn workspace @se-2/nextjs dev`, then on SIGINT/SIGTERM called `child.kill()` on the top-level yarn. Yarn doesn't forward signals to its descendants, so the outer yarn exited but left `hardhat node` (two `yarn` layers deeper) reparented to launchd and running forever.

Fix: spawn both children with `detached: true` (new process-group leader) and swap `child.kill()` for `process.kill(-child.pid, signal)` to signal the whole group. One transparent knock-on: detached children aren't in the terminal's foreground group anymore, so they don't receive Ctrl+C directly — but the launcher's SIGINT handler already fans signals out via `killTree`, and neither child reads stdin, so the end-to-end behavior is unchanged for the user.

### 2026-04-20 — [claude] Removed `web3protocol` preview path in `FileBrowser.tsx`

On devnet (app hosted at `*.nip.io` / eth.limo origin), clicking a file preview triggered Chrome's **Local Network Access** permission prompt ("Access other apps and services on this device — Block / Allow"). Root cause: `FileBrowser.fetchFileContent` dynamically imported `web3protocol`, which bundles WASM, constructs its own `Client` with an RPC URL from `NEXT_PUBLIC_HARDHAT_RPC_URL`, and fetches outside wagmi's configured transport. That out-of-band fetcher crosses the public→private-IP boundary on first preview click and trips LNA, which users read as malware and bounce from.

Fix: deleted the web3protocol attempt entirely; the direct `publicClient.readContract` + gateway path was already the fallback and covers both on-chain SSTORE2 chunks (via `web3-next-chunk` pagination) and `message/external-body` delegation for IPFS/Arweave/HTTPS mirrors. The web3protocol branch also collapsed duplicate Content-Type headers (breaking external-body detection) and fell through on empty bodies anyway — strictly extra surface area. If a "native transport helper" (opt-in local IPFS node, etc.) is added later, it must be an explicit user toggle, never automatic from a preview click. Package `web3protocol` left in `package.json` for now; tree-shakeable and can be removed in a cleanup pass.

### 2026-04-19 — [claude] Folder child-count renders as "Folder" in lens mode
